# nodestorm v0.7 — terminal-theme palettes, light/dark modes, system auto-follow

Approved 2026-07-17. Single release on `claude/terminal-themes-color-pallets-f92d7c`,
one PR to `main`, version-bump commit at the end.

## Theming model

Twelve theme **families**, each defining a **dark and a light variant**, plus a
**mode** setting `auto | dark | light` where `auto` live-follows the OS. The
preference is **global** (all sessions), persisted to `preferences.json` in the
platform data dir, and never enters `SessionState` — so it is structurally
excluded from undo snapshots, MCP payloads, and exports.

Families (id → display name): `nodestorm` → Nodestorm (default; light side
designed for this release), `solarized` → Solarized, `gruvbox` → Gruvbox,
`catppuccin` → Catppuccin (Mocha/Latte), `nord` → Nord (light = Snow Storm,
derived aurora), `dracula` → Dracula (light = Alucard-inspired, derived),
`tokyo-night` → Tokyo Night (Night/Day), `one` → One (One Dark/One Light),
`github` → GitHub (Primer), `everforest` → Everforest (medium),
`rose-pine` → Rosé Pine (Main/Dawn), `monokai` → Monokai (light derived).

## CSS mechanism (`assets/main.css`)

- The 16 existing `:root` tokens stay byte-identical as the no-support
  fallback; five new tokens join them: `--on-accent`, `--on-badge`,
  `--shadow` (color only — alpha stays at usage sites via the existing
  `color-mix(… N%, transparent)` idiom), `--dot-grid`, and root-only derived
  `--mini-viewport-fill: color-mix(in srgb, var(--text) 8%, transparent)`.
  The seven stray literals (lines 153, 555, 739, 791, 885, 992, 1113) are
  rewritten onto these tokens.
- One flat block per family: `[data-theme="<id>"] { --bg: light-dark(L, D); … }`
  defining all 20 per-family tokens (light argument first). All twelve blocks
  live inside a single `@supports (color: light-dark(#000, #fff))` guard so
  engines without `light-dark()` (WebKitGTK < 2.46) drop the overrides
  wholesale and land on the intact dark fallback.
- Mode is pure `color-scheme`, outside the guard:
  `[data-mode="dark"] { color-scheme: dark; }`,
  `[data-mode="light"] { color-scheme: light; }`,
  `[data-mode="auto"] { color-scheme: light dark; }` — `light-dark()` then
  resolves per OS preference in auto, including live OS flips (WebView2 keeps
  `PreferredColorScheme = AUTO`; dioxus never overrides it).
- `div.app` carries `data-theme`/`data-mode` declaratively and gains
  `background: var(--bg); color: var(--text);` — `html,body` resolve those
  vars against the un-themed `:root`, and `color` inherits as a computed
  value, so the re-anchor at the attribute-bearing element is required.
- Light-variant `--shadow` values carry their own alpha (8-digit hex, `…59` =
  35%) so effective shadows land at ~12-16% in light mode vs 35-45% in dark
  with zero usage-site changes.

## Rust surface

- **`src/theme.rs`** (pure, no UI deps): `ThemeFamily { id, name }`,
  `FAMILIES: [ThemeFamily; 12]`, `DEFAULT_FAMILY = "nodestorm"`,
  `REQUIRED_TOKENS: [&str; 20]`, `Mode { Auto (default), Dark, Light }`
  (serde lowercase, `as_str()`), `family(id) -> Option<&'static ThemeFamily>`.
  Colocated integrity tests `include_str!` the stylesheet and assert: every
  family id has a `[data-theme]` block defining all 20 tokens via
  `light-dark(`, no orphan blocks, `:root` defines every token, the mode
  rules set `color-scheme`, and the family blocks sit inside the `@supports`
  guard. `cargo test` alone keeps registry and CSS in lock-step.
- **`src/prefs.rs`**: `Preferences { version: u32 = 1, theme: String,
  mode: Mode }`; `default_prefs_path()` → `preferences.json` next to the
  session data (`ProjectDirs "dev"/"nodestorm"/"nodestorm"`);
  `load_or_default(path)` — missing → default, corrupt → moved aside to
  `.json.corrupt` + default (persist.rs pattern), unknown theme id →
  `DEFAULT_FAMILY` with mode preserved; `save(path, prefs)` via
  `persist::write_atomic` (promoted to `pub(crate)`).
- **`src/cli.rs`**: `--prefs <file>` override + `prefs_path()`, mirroring
  `--session`/`session_path()`. Required so E2E never touches the real
  user's preferences.

## UI

- `ui/mod.rs`: `ThemePref(pub Signal<Preferences>)` context newtype;
  `tao_theme(Mode) -> Option<tao::window::Theme>` (`Dark→Some(Dark)`,
  `Light→Some(Light)`, `Auto→None`); `launch()` loads prefs before building
  the window, applies `WindowBuilder::with_theme(tao_theme(prefs.mode))` so
  the native title bar is correct from first paint, and passes the loaded
  value via `.with_context(prefs)`.
- `app.rs`: provides the `ThemePref` signal seeded from context; the root
  `div.app` gains `"data-theme"`/`"data-mode"`; a `use_effect` calls
  `window.window.set_theme(tao_theme(mode))` on every mode change so the
  **native title bar follows the in-app mode live** (tao 0.34.8
  `Window::set_theme`, DWM immersive dark mode on Windows).
- New `src/ui/theme_menu.rs`: `ThemeMenu`, a third topbar dropdown copying
  the Export-menu pattern (`use_signal` open flag, `.menu-catcher`,
  `.export-dropdown theme-dropdown`). Trigger text `Theme ▾`. Contents: a
  3-button mode row **Auto / Light / Dark** (mode clicks do NOT close the
  menu) and twelve family rows — display name, 4-dot swatch strip, `✓` on
  the active family (family clicks close the menu). Swatches use the
  local-scope trick: a `span { "data-theme": id }` wrapper re-scopes the
  tokens, so dots on `var(--bg)/var(--accent)/var(--status-modified)/var(--status-affected)`
  preview each family in the currently-resolved mode. Every click mutates
  the signal then saves via `cli.prefs_path()` + `prefs::save`
  (`tracing::warn!` on failure).
- No keyboard shortcut (the keymap is canvas-focus-only — poor fit for a
  global preference).
- Exports stay theme-independent: `export.rs` hexes still mirror the
  `:root` fallback, so its "mirrors `--status-*`" comment stays true.

## Palette data

The full paste-ready CSS blocks (all 12 families × 20 tokens,
`light-dark(light, dark)`) live in the implementation plan
(`docs/superpowers/plans/2026-07-17-v0.7-theming.md`) and are the spec for
Task T2. Sources are the official palettes; values marked *derived* are
designed for this app (intermediate bg-ladder steps; nodestorm-light;
nord-light aurora; Alucard-inspired Dracula light; monokai-light). Minor
visual tuning is allowed with screenshot evidence; token names/structure are
not negotiable.
